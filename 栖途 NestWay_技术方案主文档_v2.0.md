# 1. 产品概述与核心功能

# 2. 终端形态选型

## 2.1 选型结论：UniApp 微信小程序（MVP 唯一终端）

综合团队规模、开发周期和资源限制，MVP 阶段放弃 App 主端，全面转向 UniApp 微信小程序。

  ----------------------------------------------------------------------------
  **终端**            **原方案**               **新方案（MVP）**
  ------------------- ------------------------ -------------------------------
  iOS / Android App   主端，承载全部 P0 功能   ❌ 暂不开发，资源不足

  微信小程序          轻量获客入口             ✅ MVP 唯一终端，承载全部功能

  H5 网页             不开发                   ❌ 不开发
  ----------------------------------------------------------------------------

## 2.2 选择小程序而非 App 的理由

> 零安装门槛：用户扫码即用，冷启动成本极低，适合 MVP 快速验证
>
> 开发效率高：UniApp 一套代码，Vue 3 语法，团队上手快，无需维护双端原生代码
>
> 分发成本低：无需上架 App Store / 华为市场，规避审核周期风险
>
> 社交传播强：女性用户出行前相互分享目的地安全卡片，天然传播路径

## 2.3 小程序的已知限制与应对策略

# 3. 前端框架选型

## 3.1 选型结论：UniApp + Vue 3 + TypeScript

小程序端使用 UniApp 框架，基于 Vue 3 Composition API + TypeScript 开发，一套代码编译为微信小程序，保留未来扩展至支付宝小程序或 H5 的可能性。

  -----------------------------------------------------------------------------------------
  **对比维度**      **UniApp + Vue 3**      **原生微信小程序开发**   **Taro（React）**
  ----------------- ----------------------- ------------------------ ----------------------
  学习曲线          低（Vue 3 语法）        中（需学 WXML/WXSS）     中高（需熟悉 React）

  跨平台能力        强（一套代码多端）      无（仅微信）             强

  生态与组件库      uni-ui、uView UI 成熟   微信原生组件             NutUI 等

  TypeScript 支持   完整支持                有限                     完整支持

  团队适配度        高（Vue 技术栈）        中                       低
  -----------------------------------------------------------------------------------------

## 3.2 推荐 UI 组件库：uView UI 2.0

uView UI 2.0 是专为 UniApp 设计的组件库，支持 Vue 3，内置丰富的移动端组件（按钮、表单、弹窗、地图等），可显著减少 UI 开发工作量。

## 3.3 状态管理：Pinia

使用 Pinia 替代 Vuex，语法更简洁，与 Vue 3 Composition API 配合更自然，适合管理用户信息、护送任务状态、AI 对话上下文等全局状态。

# 4. 后端框架选型

## 4.1 选型结论：Node.js + Express + TypeScript

MVP 阶段后端改用 Node.js + Express + TypeScript，放弃原方案的 Spring Boot。理由如下：

> 团队基础匹配：后端同学有 Node.js 实际使用经验，上手成本极低，可直接开发
>
> 语言统一：前后端均使用 TypeScript，接口类型定义可共享，减少沟通成本
>
> 启动成本低：无需配置 JDK、Maven 等环境，npm install 即可启动
>
> AI 接口调用友好：Node.js 对流式 HTTP 响应（SSE）处理更简洁，适合 AI 对话流式输出
>
> 部署简单：一台 2 核 2G 轻量云服务器即可跑起来

## 4.2 模块划分

  -------------------------------------------------------------------------

# 6. 核心功能技术方案

## 6.1 虚拟护送 --- 前台保活 + 定时提醒（降级方案）

由于微信小程序后台定位稳定性不足，MVP 阶段虚拟护送采用降级方案：

前端实现：

> 护送开始后，小程序保持前台运行，使用 wx.setKeepScreenOn() 开启屏幕常亮
>
> 每隔 30 秒通过 uni.getLocation() 获取当前位置并上报后端，同时刷新 Redis 心跳 TTL
>
> 前台展示倒计时 UI，用户可随时点击「安全到达」主动结束护送
>
> 若用户切换到后台，前台弹出提示「护送进行中，请保持小程序在前台运行」

后端实现：

> 创建护送任务时写入 Redis Key，TTL = 预计时长 + 缓冲时间（+5 分钟）
>
> Redis Key 过期后触发超时事件，向紧急联系人发送微信服务通知 + 短信双通道告警
>
> 告警内容包含：用户姓名、最后已知坐标、高德地图定位链接

降级说明：此方案要求用户在护送期间保持小程序前台运行，无法实现后台静默追踪。后续 App 版本上线后，将恢复完整的后台定位能力。

## 6.2 SOS 一键求助 --- 三路并发触发

触发逻辑（前端）：

> 用户长按 SOS 按钮 3 秒（防误触），震动反馈 + 可取消倒计时 UI
>
> 确认后并发执行三路操作：
>
> ① 拨号：调用 wx.makePhoneCall() 调起拨号界面（需用户点击确认）
>
> ② 上报：调用 SOS API 上报事件 + 当前坐标，后端异步推送通知给紧急联系人
>
> ③ 分享：通过 wx.shareFileMessage 将位置链接发送给联系人

后端处理：SOS API 异步触发通知，向所有紧急联系人发送微信服务通知 + 短信双通道告警。SOS 事件全量持久化，供用户查看历史记录。

## 6.4 AI 实时对话 --- SSE 流式输出架构

前端通过 SSE（Server-Sent Events）连接后端，发送用户消息后接收流式 Token 逐字输出（打字机效果）。UniApp 小程序通过 uni.request 的 enableChunked 参数支持分块接收，实现流式效果。

后端 router/ai 模块调用 LLM API（具体模型待刘璇调研结果确定），将流式响应转发给前端。对话上下文：短期存 Redis（滑动窗口，保留最近 10 轮），长期持久化至 MySQL。系统 Prompt 预设「女性独旅安全顾问」角色。
